# AGENTS.md

## Cursor Cloud specific instructions

DivLab is a Next.js 16 (App Router, Turbopack) + Supabase app; the UI is in Swedish. Standard scripts live in `package.json` (`dev`, `build`, `start`, `lint`).

### Services

- **Next.js dev server** — `npm run dev` (http://localhost:3000). This is the app.
- **Supabase** (Postgres + Auth + Storage) — required for anything beyond the static marketing pages. Auth, profiles, forum and private messages all use Supabase. Runs locally via the Supabase CLI (Docker).

### Starting the backend (the update script does NOT do this)

Docker and the Supabase CLI are pre-installed in the VM image, but nothing auto-starts. Bring the stack up before running or testing the app:

1. If `docker info` fails, start the daemon: `sudo dockerd &`, then `sudo chmod 666 /var/run/docker.sock` (lets the non-root user use Docker). The daemon is already configured for this VM (`/etc/docker/daemon.json` uses fuse-overlayfs with the containerd snapshotter disabled; iptables is set to iptables-legacy).
2. From the repo root: `supabase start` (applies everything in `supabase/migrations` and runs `supabase/seed.sql`). Supabase Studio is at http://localhost:54323 and Mailpit (auth emails) at http://localhost:54324.

### Required environment variables

The app reads `NEXT_PUBLIC_SUPABASE_URL` and `NEXT_PUBLIC_SUPABASE_PUBLISHABLE_KEY` (see `lib/supabase/config.ts`), and the middleware in `proxy.ts` runs on every route and throws when they are missing — so **every** page 500s without them.

**Local development only.** Create a `.env.local` at the repo root (gitignored). Do not hardcode or commit Supabase credentials in this file or anywhere else in the repo.

**Production and staging** credentials are configured in the deployment platform (for example Vercel environment variables). They are separate from local development and must never be copied into `AGENTS.md`, committed files, or agent instructions.

To populate `.env.local` for local work:

1. Start the local Supabase stack first (see "Starting the backend" above).
2. From the repo root, run `supabase status`.
3. Copy the reported `API URL` into `NEXT_PUBLIC_SUPABASE_URL` and the `Publishable key` into `NEXT_PUBLIC_SUPABASE_PUBLISHABLE_KEY`. (`supabase status` may print JSON with `API_URL` and `PUBLISHABLE_KEY` — use those same values.)
4. Set `NEXT_PUBLIC_SITE_URL=http://localhost:3000`.

Example `.env.local` shape (fill in values from `supabase status`, not from this document):

```
NEXT_PUBLIC_SUPABASE_URL=<API URL from supabase status>
NEXT_PUBLIC_SUPABASE_PUBLISHABLE_KEY=<Publishable key from supabase status>
NEXT_PUBLIC_SITE_URL=http://localhost:3000
```

Re-run `supabase status` after `supabase db reset` or if the local stack is restarted and keys change.

### Non-obvious gotchas

- **`supabase/seed.sql` is required for local dev — do not delete it.** The `20260707181000_create_profiles` migration relies on the default table grants that hosted Supabase gives the `anon`/`authenticated` roles, but the local CLI's `postgres` role does not grant `SELECT/INSERT/UPDATE` on new public tables. Without the seed, `profiles` reads fail with `permission denied for table profiles`, which breaks the dashboard and every logged-in page. `seed.sql` restores exactly the grants the profile RLS policies already assume. Every other table ships explicit grants inside its own migration; only `profiles` depends on this. The file is local-only (it is not applied by `supabase db push`).
- Local email confirmation is disabled (`enable_confirmations = false` in `supabase/config.toml`), so `/register` signups are auto-confirmed and log in immediately — no email step is needed for testing.
- Reset local DB state with `supabase db reset` (re-runs migrations + seed and wipes all data/users).
- `supabase/config.toml` is committed local-dev config generated by `supabase init`; it is not used in production.
